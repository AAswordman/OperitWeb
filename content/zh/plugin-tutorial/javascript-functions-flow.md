# 函数、模板字符串与流程控制

这一章要解决的是：你已经会放变量了，但一个真正的工具函数到底是怎么把输入变成输出的？

## 学完这一章，你应该能做到

- 理解“一个插件工具本质上就是一个函数”。
- 看懂参数、返回值、模板字符串、`if`、循环在脚本里的最常见写法。
- 能写出一个最小但结构清楚的业务函数。

## 建议先具备

- 已经看过《变量、对象与数组》。
- 对对象字段访问不陌生。

## 本章对应的真实文件

- `examples/quick_start.ts`
- `examples/github/src/index.ts`
- `docs/SCRIPT_DEV_GUIDE.md`

## 先看一个最小例子

```js
function buildSummary(params) {
  const lines = params.lines || [];
  const max = params.max || 2;

  let result = "";

  for (let i = 0; i < lines.length && i < max; i += 1) {
    result += `第 ${i + 1} 行：${lines[i]}/n`;
  }

  if (!result) {
    return "没有可展示的内容";
  }

  return result.trim();
}
```

## 把例子拆开理解

### 函数就是工具的基本单位

最先要建立的认知是：  
宿主不是“直接执行你整份脚本文件里的任何代码”，而是通过导出的函数来调用你定义的工具。

```js
function buildSummary(params) {
  return "...";
}
```

在后面的章节里，这样的函数会进一步变成：

- 有 `async`
- 有错误处理
- 最终通过 `complete(...)` 回传结果

但它的本质仍然是“一个接收参数、处理参数、返回结果的函数”。

### 参数通常是一个对象

```js
function buildSummary(params) {
  const lines = params.lines || [];
  const max = params.max || 2;
}
```

为什么插件里喜欢把参数写成对象？

- 字段名字更清楚
- 参数可以逐步扩展
- 和 `METADATA.parameters` 的描述方式天然一致

### 模板字符串让结果拼装更顺手

```js
result += `第 ${i + 1} 行：${lines[i]}/n`;
```

模板字符串非常适合：

- 拼消息文本
- 拼日志
- 拼返回摘要

写法就是反引号加 `${...}`。

### `if` 用来做输入分支判断

```js
if (!result) {
  return "没有可展示的内容";
}
```

插件里最常见的分支判断包括：

- 参数缺失
- 文件不存在
- 搜索结果为空
- 某一步执行失败

### 循环常用于批量处理结果

```js
for (let i = 0; i < lines.length && i < max; i += 1) {
  result += `第 ${i + 1} 行：${lines[i]}/n`;
}
```

插件里常见的循环对象有：

- 文件列表
- UI 节点列表
- 搜索结果列表
- 消息数组

## 回到源码仓库，它为什么这样写

你去看 `examples/github/src/index.ts` 时，会发现包再大，本质还是同一套模式：

- 每个工具都是一个函数
- 工具接收结构化参数
- 业务逻辑内部会做分支处理
- 最后把结果组织成对象或文本再返回

只是工程更大以后，函数会被拆到不同文件里，入口文件负责聚合导出。  
所以这一章虽然很基础，但其实是在给后面的工程化章节打底。

## 本章最容易踩的坑

### 坑 1：函数里什么都想做

如果一个函数既解析输入、又读文件、又做网络请求、又拼接所有输出，后面就会很难维护。  
哪怕是最小插件，也应该先形成“输入 -> 处理 -> 输出”的清晰流程。

### 坑 2：字符串拼接全靠 `+`

简单场景当然能用，但一旦有变量、有换行、有多个字段，模板字符串会更可读。

### 坑 3：参数不是对象，而是一串松散位置参数

这在插件开发里很快就会变得难维护，因为宿主侧、`METADATA`、文档页都更适合描述命名参数。
