# 10. 工具：MCP

上一节我们讲了沙盒包，这一节继续讲另外一种工具形式。

MCP 全称是 *Model Context Protocol*，是一种以*单独进程启动*的工具服务器。运行非常的吃资源。在软件中，我们以工具包的形式，将其兼容进现有工具体系，和前面介绍的沙盒包**完全等价兼容**。同样也是需要动态激活，然后再去调用里面的工具。

由于mcp运行非常吃资源，因此软件会**动态加载**以及**动态卸载**mcp，并在需要调用的时候再次开始该服务器。


在 AI 看来，MCP、沙盒包、Skill 这三种东西是**没有区别的**。AI 都用 `use_package` 去激活，然后用 `package_proxy` 去调用里面的工具。你不需要告诉 AI 这个功能来自哪种包，它自己通过**统一的接口**就能用。

## 安装 MCP

在软件里点开「包管理」，切到 MCP 那一栏，点右下角的加号或者「导入/连接」按钮，会看到一个弹窗，里面有好几种导入方式。

### 配置导入

适合 `npx`、`uvx` 这种一条命令就能跑的服务。打开「配置导入」的页面，把 MCP 的 JSON 贴进去，点「**合并配置**」就行了。

![配置导入界面](</manuals/assets/tools/mcp_config_import.jpg>)

比如说你要加一个 Fetch 的 MCP，粘贴这样一段 JSON 就行：

```json
{
  "mcpServers": {
    "fetch": {
      "command": "uvx",
      "args": ["mcp-server-fetch"],
      "env": {},
      "autoApprove": []
    }
  }
}
```

里面的 `fetch` 就是*插件 ID*，建议只用英文字母、数字和下划线，尽量小写。合并成功后，插件就会出现在列表里。

**需要特别强调，这种方式只支持本地stdio运行的mcp，远程mcp请参考后面的专用远程导入，一定不要用这里的配置合并导入！！**

接着，点击右边的运行按钮，完成导入。

### 从仓库导入

适合你拿到的 MCP 是一个 GitHub 上的项目，比如用 Python、Node.js 或 TypeScript 写的。切到「仓库」页面，填上仓库链接，再给个插件名称，点获取 MCP 之后点导入就行。

![仓库导入界面](</manuals/assets/tools/mcp_repo_import.jpg>)

这里要特别注意的是，软件对于这种直接给源码的形式，目前*只做了node以及python*的自动识别和部署。如果仓库不规范/不是这两种情况，请不要使用从仓库导入。

### 从 ZIP 导入

适合你已经有一个本地的插件压缩包。这个的原理和上面从仓库导入是一样的。

### 连接远程服务

适合你已经在电脑或服务器上把 MCP 跑起来了，手机只管连过去用。切到「远程」页面，填上服务地址（比如 `http://192.168.1.100:8752`），选连接类型（`httpStream` 或 `sse`）。**不要填错类型！！！**

![远程连接界面](</manuals/assets/tools/mcp_remote_connect.jpg>)

如果远程服务需要鉴权，可以填 *Bearer Token*。如果需要额外的请求头，下面还可以添加自定义请求头。保存后它会以远程插件的身份出现在列表里，**不需要部署**，连接成功就能用。

记得这个配置好也要点启动按钮。

需要说明的是，我们的远程mcp的配置，主要记录在 `pluginMetadata` 里，这点和当前通用的不同，因此**不能使用配置合并去导入**。


### 手动编辑配置文件

这就是*最通用的方式*了。这个过程比较麻烦，你也可以直接丢给Operit的默认ai去处理这件事，ai会自动完成配置的修改以及插件的启动。

操作过程和Linux/Windows配置mcp类似。MCP 的配置目录在 `/sdcard/Download/Operit/mcp_plugins/` 路径下，主要的配置文件是 `mcp_config.json`。你可能会注意到还有一个 `server_status.json`，那个只是**状态缓存**，不用管它。

`mcp_config.json` 的结构这里不再介绍，不懂的可以自行搜索。

有一点需要特别注意：MCP 的启动工作目录是固定的 Linux 路径 `~/mcp_plugins/<插件ID最后一段>/`，所以 `args` 里的路径要写*相对于这个目录*的路径。你可以理解为，此过程等价于，先cd到这个固定的路径，然后在这个路径下完成启动命令。

这个路径的 `~` 指的是*软件内置Linux下的路径*，而不是写 Android 那边的路径。

那么问题就来了，为什么我们是在sdcard下放的文件，写的配置文件，但是启动路径默认居然是在linux里面呢？原因也很简单，软件在部署的时候，会把sdcard里面的mcp，**复制一份到linux环境**里面去，接着继续在这个环境里面启动。这一点尤其要注意。而上述这个过程也就是所谓的**部署过程**。

对于 `npx` 类型的 MCP，还有一点要说明：配置里还是按常规写 `"command": "npx"`，但软件内部实际启动的时候会*自动把它转成 `pnpm dlx`* 来执行。所以 Linux 环境里需要装 `pnpm`，不要手动改成 `npm` 或 `pnpm`，改了反而可能有兼容问题。

## 部署原理

承接着这个配置，我们继续讲。并不是所有的mcp都是成品出来的，很多都是文件里面，一堆py，一堆ts，*依赖还得下*。

软件有一套**自动识别**的机制，能够完成上述复制机制的同时，自动下载依赖，自动配置环境。这里需要注意，目前*只支持node和py*，当然别的语言也不是没有办法，只不过需要写自定义部署命令了。当然最方便的还是直接丢给ai，agent出手直接解决。

未部署的插件在linux下*暂无复制文件*，根据这个特点，我们为启动按钮的启动前加了一个检测，如果插件没有部署则**自动部署**。因此现在不太需要为手动点部署操心了。

## 启动原理

说了这些导入方法，大家也有一个印象了。那么问题来了，点击启动按钮，Operit到底做了什么？出现问题咋回事？让我们从原理开始看。

软件会先在终端开始*三个界面*: `local`，这个是默认会话。`mcp_shared`，软件会在这个会话里面探测pnpm环境是否搞定，以及是否可以正常复制bridge。如果都没有问题，那么就会再新开第三个窗口，在里面*单独启动mcp bridge*。当然，这同时也意味着，**如果出现了bug，就可以去终端界面的第三个窗口去排查**。

启动完成后，软件会尝试继续在bridge注册每一个插件。如果注册成功。那么就会再次发送工具探测，探测成功就会关闭mcp。那么这里就有几个关键点:首先是这mcp要正常启动，如果缺失环境，就会报错退出。其次就是，当手机*运行内存不够*的时候，可能终端运行mcp会被杀，可以退一点程序防止double free的报错。具体的问题具体分析，要看报错来，有的报错则是说依赖没装，这个时候就得为这个报错补依赖。

当下一次use package的时候，软件会*再次激活*这个mcp，并可能在几分钟后关闭。这是为了确保同一时间正在运行的有效mcp数量不至于太多。

正如前面手动编辑配置文件部分所说，启动的具体过程，是跟着配置走的。*东西最好一点一点加*，特别是mcp太重量级了，远程还好一些。

有些功能和包管理内置重复/和skill重复的，尽量用这两个，**轻量一些**，而非mcp。

## MCP简介

这个时候我们会发现一件事: 按照我们前面的普通沙盒包的理论，ai需要有简介和包名，才知道去激活包啊，但是到目前为止mcp都还没有简介呢。是的，所以在启动完成之后，我们还有最后一个步骤:**生成简介**。完全可以猜到的是，生成简介的这个步骤是拿*功能模型*完成的，生成依据是前面启动完成拉到的工具和工具介绍。所以这个*功能模型如果不配好*，那么，这一步的简介基本上也是炸掉的，注意排查。

## 常见问题

MCP 用不起来的时候，大部分情况下都是以下几类问题：

**路径写错了**：这是最容易犯的错。`args` 里写了 Android 路径（比如 `/sdcard/...`），但启动工作目录是 Linux 那边的 `~/mcp_plugins/`，对不上当然跑不起来。

**依赖没装**：比如 node 类的 MCP 需要 `pnpm`，Python 类的需要 `uv` 或 `pip` 装好依赖。在 Linux 终端里检查一下这些依赖是不是齐全。

**环境变量没填**：有些 MCP 需要 API Key 之类的环境变量，写在 `env` 字段里。漏掉了或者填错了，MCP 能启动但功能用不了。

**目录不存在**：插件部署到 Linux 侧的目录是自动创建的，但如果自动分析构建命令失败了，目录可能是空的。这种情况一般发生在重置终端之后，只要重新部署一下插件就可以了。
