# Operit 插件市场审核规范

## 目的

人工审核只处理系统无法可靠判断的内容：项目是否真实有用、改动是否有意义、是否存在风险，以及是否有明显的幻觉或欺骗性内容。

版本号、必填字段、安装配置、资产格式和其他可自动判定的问题，由发布系统负责拦截。审核人不需要重复检查这些显而易见的基础项。

审核对象是具体版本。同一项目的多个待审版本应分别处理。

## 审核接口

审核接口均使用管理员或审核员令牌：

```http
Authorization: Bearer <admin_token>
```

也可以使用请求头 `x-operit-admin-token`。所有接口的基础地址为 `https://api.operit.app`。

### 读取待审队列

```http
GET /market/v2/admin/review/entries?stateCode=pending&limit=50&offset=0
```

`items[]` 的每一行代表一个待审版本，而不是一个项目。以 `item.id` 作为 `entryId`，以 `item.version.id` 作为 `versionId`；同一项目有多个待审版本时必须分别审核。

### 读取项目审核详情

```http
GET /market/v2/admin/review/entries/{entryId}
```

返回项目资料、`versions[]`、仓库来源或托管资产。审核原作者更新时，用它确认是否已有通过版本及版本发布者；审核换作者更新时，用最近通过版本作为对比基线，再拉取 API 返回的仓库来源或资产进行比较。

### 提交结论

```http
POST /market/v2/entries/{entryId}/review/approve
Content-Type: application/json

{ "entryId": "{entryId}", "versionId": "{versionId}" }
```

```http
POST /market/v2/entries/{entryId}/review/changes
Content-Type: application/json

{ "entryId": "{entryId}", "versionId": "{versionId}", "reasonCode": "ai-hallucination" }
```

```http
POST /market/v2/entries/{entryId}/review/reject
Content-Type: application/json

{ "entryId": "{entryId}", "versionId": "{versionId}", "reasonCode": "security-risk" }
```

`entryId` 必须与 URL 一致，且 `versionId` 必须属于该项目。`changes` 与 `reject` 必须携带原因码。已上架项目审核新版本时，操作只作用于目标版本；首次发布的审核结论同时作用于项目和目标版本。

## 总则

- 不用代码量、界面复杂度、是否由 AI 辅助创作或作者知名度判断质量。
- 小功能只要真实、好用且有明确价值，就可以通过。例如只在输入框上增加输入锁的脚本，虽然实现简单，但能解决具体问题，也值得上架。
- AI 生成不是拒绝理由；**明显幻觉、虚构功能或不可用内容**才是问题。
- 已通过作者的历史记录可以降低常规更新的审核成本，但不能掩盖恶意破坏或新增风险。
- 命令拼接、Shell 调用、高权限调用、后台进程或较宽的文件访问范围，本身属于小风险，不单独作为要求修改或拒绝的理由。它们在功能需要且行为可理解时可以通过。
- 只有存在明确的恶意行为、隐蔽的数据外传、无关的破坏性操作、后门，或风险明显超出功能需要时，才以安全问题拦截。

## 审核方式

### 1. 原作者的后续版本

同一项目、同一作者，且此前已有通过版本时，默认快速通过。

只要没有明显的恶意破坏、投毒、隐私侵害、无关内容替换或其他新增高风险行为，就可以通过。常规的功能增加、修复、重构、文案调整和小范围体验优化，无需因为功能小或改动不大而卡住。Shell、高权限调用或命令拼接等实现风险，若服务于声明功能且未发现恶意用途，也按可通过处理。

出现异常信号时再深入查看，例如把原有功能替换为空壳或广告、突然加入无关的联网/命令执行、明显破坏原有能力，或存在故意规避审核的迹象。

### 2. 其他作者提交已有项目的新版本

当最近已通过版本是原作者发布，而待审版本由其他作者提交时，必须把仓库拉下来，与最近已通过版本对比。

人工只需判断两件事：

1. 改动是否有实际意义，例如修复缺陷、改善兼容性、补齐功能或优化体验；
2. 改动带来的风险是否很小或没有明显恶意风险。

两项均满足即可通过。第三方修复版可以通过，不因提交者不是原作者而额外设限。

重点排除无意义或幻觉式改动：声称修复但没有对应实现、调用不存在的 API 或文件、只堆砌自动生成内容、用占位符冒充功能，或借小改动夹带无关的危险行为。

### 3. 首次发布

首次发布主要看三点：风险、真实性和质量。

- **风险：** 是否有明确的不合理数据外传、破坏性操作、隐蔽恶意执行或其他会伤害用户的行为。正常功能所需的 Shell、高权限、后台进程或文件访问可接受。
- **真实性：** 描述的功能是否在内容中真实存在；不要接受虚构 API、虚构能力、空壳实现或明显无法工作的内容。
- **质量：** 是否解决了明确问题，用户是否能从中获得实际价值。功能可以简单，不要求大而全。

对首次发布不要求追求复杂设计或“看上去高级”。一个边界清楚、行为可靠的小工具，通常比功能描述很大但充满幻觉的项目更应该通过。

## 结论

### 通过

改动有意义，内容真实，风险小或没有风险时通过。

### 要求修改

内容本身可能有价值，但存在明显幻觉、描述与实现不符或需要作者补正的问题时，要求修改后重新提交。

### 拒绝

内容不可用、明显虚构、风险不可接受、恶意破坏、重复灌水或违反平台规则时拒绝。

审核意见只写实际发现的问题即可，不用重复记录审核路径、版本信息或系统已经自动校验的项目；这些信息由系统留存。

本规范优先保障用户安全和市场内容真实性，同时为真实、有用的创作保留尽可能低的审核阻力。
